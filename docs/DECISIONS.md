# Decisiones de arquitectura

Registro permanente de decisiones aprobadas. Solo se agregan filas, nunca
se eliminan.

| Fecha | Decisión | Motivo |
|---|---|---|
| 2026-07-13 | No implementar `freeze_panes` en la hoja "Resumen COMBOS" | La hoja tiene 5 tablas apiladas verticalmente con encabezados en filas distintas; congelar una fila fijaría el encabezado de una tabla mientras se scrollea otra, mostrando la columna equivocada fija en vez de ayudar. |
| 2026-07-13 | No optimizar la complejidad O(n²) de `dominio/preponderancia.marcar_superadas` ni el estilizado celda-por-celda de Excel | A la escala real del dominio (decenas a un par de cientos de combinaciones por corrida) el costo es imperceptible; optimizar agregaría complejidad sin beneficio medible. |
| 2026-07-13 | Mantener el easter egg de `cli/consola.py` (activado solo en la selección de reglamento) | Decisión del autor, no un descuido. Se reubicó la lógica de activación a una función propia (`pedir_seleccion_de_archivo`) para que `pedir_input` no tenga comportamiento oculto, sin eliminar la funcionalidad. |
| 2026-07-13 | Diferir la migración de objetos de dominio (dict → `dataclass`) a una sesión aparte | Cambio de mayor alcance (toca `dominio/`, `infraestructura/exportador.py` y tests). Condición de activación y detalle en `ROADMAP.md`. |
| 2026-07-13 | Bajar `requires-python` de `>=3.13` a `>=3.12` | Ampliar compatibilidad para usuarios que todavía están en 3.12 (soportada hasta octubre 2028). El proyecto no usa ninguna feature exclusiva de 3.13, y las dependencias (`openpyxl`, `pyyaml`, `rich`) corren en 3.12. |
| 2026-07-13 | Concentrar todos los objetos de dominio en un único módulo `dominio/modelos.py` en vez de un archivo por dataclass | Las cinco clases son chicas (≤10 campos) y se usan siempre en conjunto en el pipeline; un archivo único evita cinco imports por cada consumidor sin ocultar responsabilidades (el módulo tiene una única razón: definir la forma de los objetos que circulan por el dominio). |
| 2026-07-13 | Usar dataclasses mutables (no `frozen=True`) para los objetos de dominio | Los campos de marcado de `Combinacion` (`es_duplicada`, `esta_superada`, `descartada_por_usuario`, `nombre`) se completan a lo largo del pipeline; congelarlos obligaría a reemplazar el objeto en cada paso (con `replace`) y complicaría el flujo sin beneficio real a esta escala. |
| 2026-07-13 | Plan de mejoras post-v1.1.0 aprobado con orden fijo: (1) YAML con parámetros, (2) refactor del estado de sesión, (3) formato `.combos`, (4) migración a Nivel 3 + decisiones de instalación, (5) GUI, (6) "duplicar reglamento como base" | El rediseño del YAML cambia el dominio que todo lo demás consume; el objeto de sesión es prerequisito común de `.combos` y de la GUI. Hacer la GUI primero obligaría a rehacerla dos veces. Evaluado con `/evaluar`, `/ui` y el agente arquitecto. |
| 2026-07-13 | Factores alternativos modelados como "parámetros del reglamento": sección `parameters` a nivel reglamento (nombre, pregunta, opciones con etiqueta, default) y referencia `{ param: X }` en los factores de las combinaciones. Extensión retrocompatible: el número fijo sigue siendo válido | El factor depende de una propiedad del proyecto (ej. destino del edificio), no de cada combinación: preguntarlo una sola vez elimina fricción e impide respuestas inconsistentes entre combinaciones (error normativo). La elección queda registrada en la exportación por trazabilidad. |
| 2026-07-13 | Formato de sesión `.combos`: JSON con `schema_version` desde el día uno; guarda inputs y decisiones del usuario (reglamento, estados, parámetros elegidos, descartes, ediciones), nunca resultados calculados — las combinaciones se regeneran al abrir; embebe una copia del reglamento usado. Prohibido `pickle` | Compatibilidad hacia adelante garantizada, reproducibilidad del cálculo aunque el YAML externo cambie después, y seguridad al abrir archivos creados por terceros. |
| 2026-07-13 | GUI de escritorio con PySide6 (Qt 6, licencia LGPL) — desvío aprobado del stack estándar (CustomTkinter / PyQt6) | El requisito central (grilla editable con pegado multi-celda desde Excel) exige Qt; CustomTkinter no tiene grilla seria. PyQt6 es GPL: obligaría a licenciar COMBOS bajo GPL para distribuirlo, incompatible con el repo MIT. PySide6 es el binding oficial de The Qt Company, mismo Qt subyacente, API casi idéntica. |
| 2026-07-13 | Estructura de la GUI: barra de menú (Archivo / Reglamentos / Ayuda) + toolbar + 3 pestañas de trabajo (Entrada, Combinaciones, Resultados) con indicador de estado por pestaña (✓ al día / ⚠ desactualizado) y banner de recálculo. Sin pestaña "Proyecto"; metadata del proyecto como diálogo. Exportación PDF diferida | Abrir/guardar son comandos de aplicación, no una etapa de trabajo: en menú/toolbar quedan accesibles siempre (Ctrl+S desde cualquier pestaña). Calca las convenciones de SAP2000/ETABS/Excel — cero curva de aprendizaje para el usuario objetivo. |
| 2026-07-13 | Regla de invalidación entre pestañas: editar el input invalida el procesamiento con aviso visible; al recalcular se preservan los descartes de las combinaciones que no cambiaron y se informa cuáles los perdieron | Evita el peor error posible: exportar resultados desactualizados creyendo que reflejan el input actual — en ingeniería estructural es un riesgo profesional, no una molestia. |
| 2026-07-13 | Las combinaciones generadas por el reglamento son inmutables. La edición del ingeniero se hace vía "duplicar y editar" (copia editable de una generada) o creando combinaciones personalizadas desde cero; ambas quedan marcadas con su origen en pantalla y en la exportación | Trazabilidad: el reporte distingue lo que dicta la norma de lo que decidió el ingeniero, y la combinación original queda visible para comparar. Editar en el lugar destruiría esa distinción. |
| 2026-07-13 | Perfiles de reglamento separados en oficiales (empaquetados con la app, solo lectura) y de usuario (carpeta propia del usuario, ej. `%APPDATA%\COMBOS\reglamentos\`), con acción "duplicar reglamento como base". El editor visual completo se difiere (condición en `ROADMAP.md`) | Impide que un tercero rompa un reglamento oficial y genere combinaciones mal ponderadas sin darse cuenta; además resuelve que escribir en `Program Files` falla siempre en una app instalada. |
| 2026-07-14 | Adoptar CI en GitHub Actions con matriz Python 3.12/3.13/3.14 (las tres versiones que declara `requires-python`): lint (`ruff`), tests (`pytest`), seguridad estática (`bandit`) y auditoría de dependencias vulnerables (`pip-audit`) en cada push y PR contra `main`. Dependabot mensual agrupado para pip y github-actions | Cerrar el bucle de calidad automáticamente en el remoto: cualquier regresión (rota compat de 3.12, aparece una CVE en `openpyxl`, un import queda desordenado) queda marcada en rojo antes de que llegue a un release. La matriz honra la promesa de compatibilidad que hace el `pyproject.toml`, en vez de dejarla como declaración sin evidencia. |
| 2026-07-14 | `pyproject.toml` como única fuente de verdad de dependencias. Se elimina `requirements.txt` | Coexistían ambos archivos declarando lo mismo (runtime deps), con el riesgo de desincronizarse. `pip install .` desde un clon cumple el mismo rol que `pip install -r requirements.txt`, sin duplicar la lista. |
| 2026-07-14 | La sección `parameters` no lleva campo `question`: el menú CLI muestra directamente el `name` del parámetro como título, seguido de las opciones etiquetadas (ajuste sobre la decisión del 2026-07-13 que incluía "pregunta") | Menos campos que redactar para quien escribe un reglamento: la etiqueta de cada opción ya es texto libre del autor del YAML y alcanza para entender qué se está eligiendo. Decisión del usuario al aprobar la implementación del ítem 1. |
| 2026-07-14 | El parámetro `L_factor` del perfil `cirsoc2005_actualizado.yaml` modela la excepción real del CIRSOC 2005: factor de L 1.0 (default) o 0.5 en las combinaciones ELU U3–U5, según destino del edificio (cocheras, reunión pública o L > 5 kN/m² vs. resto) | Se descartó la variante "1.6 vs 0.5" registrada en la memoria de sesión: la excepción del reglamento aplica sobre el factor 1.0 de U3–U5, no sobre el 1.6 de U2. Opción A aprobada por el usuario. |
| 2026-07-14 | Un parámetro declarado en `parameters` que ninguna combinación referencia produce error duro al cargar el reglamento | Un parámetro sin uso haría que COMBOS pregunte algo sin efecto en el resultado — casi siempre es un typo en la referencia `{ param: X }`. Fallar temprano con mensaje claro evita una elección normativa que el usuario cree aplicada y no lo está. |
| 2026-07-15 | Designación normativa de combinaciones como campo `name` opcional de texto libre (ej. "U3.1"), en vez del campo estructural de agrupamiento "variante" propuesto originalmente (ids tipo 1.1/1.2). Evaluado con `/evaluar` (veredicto 🔄 rediseñar) y `/ui` | El agrupamiento que se quería modelar ya tiene nombre: la numeración de la propia norma, que el ingeniero reconoce al leerla. Un campo de solo presentación no toca ids, generador ni nombrado de exportación; ids compuestos habrían cambiado el tipo del id y la lógica interna sin aportar nada más. Además deja disponible (sin forzarla) la opción futura de nombrar la exportación según la norma. |
| 2026-07-15 | `name` único global en todo el reglamento, incluso entre estados límite distintos; no vacío; cualquier escalar se convierte a texto (`name: 9.5` → "9.5"). Sin `name`, la combinación se muestra como `id N` (no `#N`, que ya significa índice de generación en las tablas de consola) | Decisión del usuario: un nombre repetido entre estados límite generaría ambigüedad justo en el campo cuyo único propósito es identificar sin ambigüedad. Las nomenclaturas reales ya son globalmente únicas por convención (prefijos U/S), así que la regla no incomoda. |
| 2026-07-15 | Validar que el `id` de cada combinación sea único dentro de su estado límite; entre estados límite puede repetirse | La lógica interna (agrupamiento del nombrado de exportación por `combinacion_base_id`) siempre asumió ids únicos por estado sin verificarlo: un duplicado fusionaba silenciosamente dos combinaciones distintas como variantes de una. Unicidad global de id rompería los YAML existentes (CIRSOC repite id 1 en ELU y ELS) sin ganar nada, porque el id nunca se muestra cruzando estados. |
| 2026-07-15 | Los parámetros del reglamento se preguntan después de leer y validar la planilla del usuario (no al cargar el reglamento), y solo los que aplican: aquellos cuyos tipos de carga referenciados tienen al menos un estado ingresado. Los que no aplican se resuelven internamente con su default, sin preguntarse ni registrarse en el encabezado del Excel | Propuesta del usuario: preguntar al cargar el reglamento obligaba a responder por factores que quizá no participan de ninguna combinación (ej. `L_factor` en un proyecto sin estados de tipo L) — fricción pura. Registrar en el Excel una "elección" que nunca ocurrió sería trazabilidad falsa; el valor default interno existe solo para mantener la invariante "aguas abajo del paso de parámetros, los factores son siempre números". |
| 2026-07-16 | Estado de sesión como dataclass `Sesion` + funciones de módulo en `dominio/sesion.py` (`procesar`, `aplicar_descartes`, `combinaciones_resultantes`, `combinaciones_superadas`), en vez de una clase con métodos o de un contenedor pasivo cuyos pasos siguiera orquestando la CLI. `exportar` recibe la sesión (10 → 5 parámetros) | Ítem 2 del plan. La dataclass "tonta" + funciones puras mantiene el estilo del dominio existente y deja el pipeline invocable por cualquier interfaz: la CLI hoy, `.combos` y la GUI después, sin recablear la lógica. Un contenedor pasivo habría pospuesto la mitad del trabajo al ítem 3. Comportamiento visible idéntico, verificado contra un baseline de exportación celda por celda. |
| 2026-07-16 | La `Sesion` conserva por separado el reglamento crudo (`reglamento_crudo`, con las referencias `{param: X}` originales) y el resuelto (`reglamento`, con los factores ya numéricos). La resolución de parámetros siempre parte del crudo, nunca del resuelto | Prerequisito del ítem 3 (`.combos`): el formato serializa inputs y decisiones, no resultados calculados, y regenera las combinaciones al abrir; sin el crudo no se puede reaplicar una elección distinta ni reproducir el cálculo si el YAML externo cambia después. Resolver siempre desde el crudo hace la operación idempotente, algo que la GUI del ítem 5 va a necesitar cuando permita re-elegir parámetros sin recargar. |
| 2026-07-16 | El `.combos` embebe el reglamento como texto YAML íntegro (string dentro del JSON), no como estructura JSON nativa. De las elecciones guarda solo `id_parametro` + `valor` (etiqueta y nombre se re-derivan del reglamento embebido al abrir); los descartes, por índice de generación, con el determinismo de la generación fijado por test | Opción A aprobada por el usuario: trazabilidad byte a byte con el archivo usado y re-validación al abrir con el pipeline completo y ya testeado de `lector_yaml`, sin escribir deserialización propia de las dataclasses de parámetros. Re-derivar las elecciones impide que un archivo editado inyecte un valor que no corresponde a ninguna opción del reglamento. |
| 2026-07-16 | Tope de 10.000 combinaciones generables por corrida (`LIMITE_COMBINACIONES` en `dominio/sesion.py`), verificado por conteo previo sin generar nada. Aplica a toda entrada: planilla propia, `.combos` de terceros y la futura GUI | Hallazgo medio de la auditoría 2026-07-16: el producto cartesiano de grupos direccionales crece multiplicativamente y un `.combos` hostil de pocos KB podía colgar la máquina al abrirse, sin confirmación del usuario. El uso real está en el orden de los cientos, así que el límite deja dos órdenes de magnitud de margen; el chequeo en `procesar` protege cualquier interfaz presente o futura sin duplicar la regla. |
| 2026-07-16 | El esquema 1 del `.combos` solo persiste sesiones completas: `sesion_a_datos` rechaza con error una sesión sin reglamento o sin estados, y la CLI ofrece guardar únicamente al cierre del flujo. El guardado de sesiones a medias (necesario para la GUI) será un incremento de `schema_version` | Inquietud planteada por el usuario al aprobar el ítem 3. La garantía vive en el dominio, no en dónde la interfaz ponga el botón "Guardar": ni la CLI de hoy ni la GUI de mañana pueden escribir un esquema 1 incompleto por accidente. Un COMBOS viejo que abra un esquema más nuevo pide actualizar el programa en vez de fallar en silencio — exactamente el escenario para el que se versionó el formato desde el día uno. |
| 2026-07-17 | La clave del mapeo `parameters` es el id del parámetro: texto libre del autor, sin convención de nombres. Evaluado con `/evaluar`; se mantiene el formato y no se agrega un campo `id` explícito | El diseño ya separa id (clave) de `name` (título en pantalla), con el mismo patrón clave-como-id de `load_types` y `limit_states`; un campo `id` explícito rompería esa consistencia y los `.combos` ya serializan `id_parametro`. La confusión detectada ("parece exigir la convención `<algo>_factor`") era documental: todos los ejemplos usaban `L_factor` sin aclarar que el id es libre. Se corrigieron README y `ejemplo_reglamento.yaml` (comentarios aclarados + segundo parámetro de ejemplo con id corto `f1`). |
