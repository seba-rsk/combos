# Decisiones de arquitectura

Registro permanente de decisiones aprobadas. Solo se agregan filas, nunca
se eliminan.

| Fecha | Decisión | Motivo |
|---|---|---|
| 2026-07-13 | No implementar `freeze_panes` en la hoja "Resumen COMBOS" | La hoja tiene 5 tablas apiladas verticalmente con encabezados en filas distintas; congelar una fila fijaría el encabezado de una tabla mientras se scrollea otra, mostrando la columna equivocada fija en vez de ayudar. |
| 2026-07-13 | No optimizar la complejidad O(n²) de `dominio/preponderancia.marcar_superadas` ni el estilizado celda-por-celda de Excel | A la escala real del dominio (decenas a un par de cientos de combinaciones por corrida) el costo es imperceptible; optimizar agregaría complejidad sin beneficio medible. |
| 2026-07-13 | Mantener el easter egg de `cli/consola.py` (activado solo en la selección de reglamento) | Decisión del autor, no un descuido. Se reubicó la lógica de activación a una función propia (`pedir_seleccion_de_archivo`) para que `pedir_input` no tenga comportamiento oculto, sin eliminar la funcionalidad. |
| 2026-07-13 | Diferir la migración de objetos de dominio (dict → `dataclass`) a una sesión aparte | Cambio de mayor alcance (toca `dominio/`, `infraestructura/exportador.py` y tests). Condición de activación y detalle en `ROADMAP.md`. |
| 2026-07-13 | Bajar `requires-python` de `>=3.13` a `>=3.12` | Ampliar compatibilidad para usuarios que todavía están en 3.12 (soportada hasta octubre 2028). El proyecto no usa ninguna feature exclusiva de 3.13, y las dependencias (`openpyxl`, `pyyaml`, `rich`) corren en 3.12. |
| 2026-07-13 | Concentrar todos los objetos de dominio en un único módulo `dominio/modelos.py` en vez de un archivo por dataclass | Las cinco clases son chicas (≤10 campos) y se usan siempre en conjunto en el pipeline; un archivo único evita cinco imports por cada consumidor sin ocultar responsabilidades (el módulo tiene una única razón: definir la forma de los objetos que circulan por el dominio). |
| 2026-07-13 | Usar dataclasses mutables (no `frozen=True`) para los objetos de dominio | Los campos de marcado de `Combinacion` (`es_duplicada`, `esta_superada`, `descartada_por_usuario`, `nombre`) se completan a lo largo del pipeline; congelarlos obligaría a reemplazar el objeto en cada paso (con `replace`) y complicaría el flujo sin beneficio real a esta escala. |
| 2026-07-13 | Plan de mejoras post-v1.1.0 aprobado con orden fijo: (1) YAML con parámetros, (2) refactor del estado de sesión, (3) formato `.combos`, (4) migración a Nivel 3 + decisiones de instalación, (5) GUI, (6) "duplicar reglamento como base" | El rediseño del YAML cambia el dominio que todo lo demás consume; el objeto de sesión es prerequisito común de `.combos` y de la GUI. Hacer la GUI primero obligaría a rehacerla dos veces. Evaluado con `/evaluar`, `/ui` y el agente arquitecto. |
| 2026-07-13 | Factores alternativos modelados como "parámetros del reglamento": sección `parameters` a nivel reglamento (nombre, pregunta, opciones con etiqueta, default) y referencia `{ param: X }` en los factores de las combinaciones. Extensión retrocompatible: el número fijo sigue siendo válido | El factor depende de una propiedad del proyecto (ej. destino del edificio), no de cada combinación: preguntarlo una sola vez elimina fricción e impide respuestas inconsistentes entre combinaciones (error normativo). La elección queda registrada en la exportación por trazabilidad. |
| 2026-07-13 | Formato de sesión `.combos`: JSON con `schema_version` desde el día uno; guarda inputs y decisiones del usuario (reglamento, estados, parámetros elegidos, descartes, ediciones), nunca resultados calculados — las combinaciones se regeneran al abrir; embebe una copia del reglamento usado. Prohibido `pickle` | Compatibilidad hacia adelante garantizada, reproducibilidad del cálculo aunque el YAML externo cambie después, y seguridad al abrir archivos creados por terceros. |
| 2026-07-13 | GUI de escritorio con PySide6 (Qt 6, licencia LGPL) — desvío aprobado del stack estándar (CustomTkinter / PyQt6) | El requisito central (grilla editable con pegado multi-celda desde Excel) exige Qt; CustomTkinter no tiene grilla seria. PyQt6 es GPL: obligaría a licenciar COMBOS bajo GPL para distribuirlo, incompatible con el repo MIT. PySide6 es el binding oficial de The Qt Company, mismo Qt subyacente, API casi idéntica. |
| 2026-07-13 | Estructura de la GUI: barra de menú (Archivo / Reglamentos / Ayuda) + toolbar + 3 pestañas de trabajo (Entrada, Combinaciones, Resultados) con indicador de estado por pestaña (✓ al día / ⚠ desactualizado) y banner de recálculo. Sin pestaña "Proyecto"; metadata del proyecto como diálogo. Exportación PDF diferida | Abrir/guardar son comandos de aplicación, no una etapa de trabajo: en menú/toolbar quedan accesibles siempre (Ctrl+S desde cualquier pestaña). Calca las convenciones de SAP2000/ETABS/Excel — cero curva de aprendizaje para el usuario objetivo. |
| 2026-07-13 | Regla de invalidación entre pestañas: editar el input invalida el procesamiento con aviso visible; al recalcular se preservan los descartes de las combinaciones que no cambiaron y se informa cuáles los perdieron | Evita el peor error posible: exportar resultados desactualizados creyendo que reflejan el input actual — en ingeniería estructural es un riesgo profesional, no una molestia. |
| 2026-07-13 | Las combinaciones generadas por el reglamento son inmutables. La edición del ingeniero se hace vía "duplicar y editar" (copia editable de una generada) o creando combinaciones personalizadas desde cero; ambas quedan marcadas con su origen en pantalla y en la exportación | Trazabilidad: el reporte distingue lo que dicta la norma de lo que decidió el ingeniero, y la combinación original queda visible para comparar. Editar en el lugar destruiría esa distinción. |
| 2026-07-13 | Perfiles de reglamento separados en oficiales (empaquetados con la app, solo lectura) y de usuario (carpeta propia del usuario, ej. `%APPDATA%\COMBOS\reglamentos\`), con acción "duplicar reglamento como base". El editor visual completo se difiere (condición en `ROADMAP.md`) | Impide que un tercero rompa un reglamento oficial y genere combinaciones mal ponderadas sin darse cuenta; además resuelve que escribir en `Program Files` falla siempre en una app instalada. |
